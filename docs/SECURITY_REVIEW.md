## ⚠️ نقاط تحتاج متابعة قبل/بعد النشر

### 1. سياسات RLS الإدارية فضفاضة
بعض السياسات تستخدم `auth.role() = 'authenticated'` بدل التحقق من الدور الفعلي (`role >= 2`):
```sql
-- الحالي (أي مستخدم مسجّل يمكنه الكتابة!):
CREATE POLICY "Admin can write config" ON app_config FOR ALL USING (auth.role() = 'authenticated');
```
**التوصية:** ربط الصلاحيات بدور المستخدم في جدول `users`. مثال:
```sql
CREATE POLICY "Only admins write config" ON app_config FOR ALL
USING (EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND role >= 2));
```
ينطبق نفس المبدأ على: تعديل المدفوعات/التبليغات/الأدوار/حظر المستخدمين (لوحة الإدارة).

> **الوضع الحالي:** لوحة الإدارة محميّة على مستوى **الواجهة** (تظهر فقط لـ `isAdmin`)، لكن لتحقيق أمان حقيقي يجب فرضها على مستوى **RLS** أيضاً.

### 2. جداول بلا سياسة UPDATE/DELETE صريحة
`payments`, `reports`, `deals` لها سياسات SELECT/INSERT فقط. عمليات التحديث (موافقة دفعة، معالجة بلاغ، إتمام صفقة) تتم من لوحة الإدارة وتحتاج سياسة UPDATE مربوطة بدور المشرف، وإلا قد تُرفض أو (إن كانت RLS متساهلة) تُسمح للجميع.

### 3. الإشعارات — صلاحية الكتابة
`notifications` عليها سياسة SELECT للمالك فقط، لكن الإدخال (إرسال إشعار لمستخدم آخر) يجب أن يتم عبر **RPC `SECURITY DEFINER`** أو Edge Function، لا مباشرةً من الكلاينت.

### 4. معرّف التطبيق وبيانات الاعتماد
- تم تغيير `applicationId`/bundle id إلى `com.sweeda.realestate` (بدل `com.example.*`).
- قبل النشر: أنشئ keystore حقيقي ولا ترفعه أبداً (انظر `BUILD_GUIDE.md`).

### 5. OTP
`otp_codes` يسمح بـ INSERT للجميع (`WITH CHECK (true)`) — منطقي لتوليد OTP، لكن تأكّد أن `verify_otp` يحدّد محاولات فاشلة/انتهاء صلاحية لمنع brute-force.

**✅ تحديث 2026-06-05:** أُضيف `generate_otp_v2` مع حدّ معدّل (5 طلبات/10 دقائق لكل identifier) + `verify_otp_v2` يحذف الكود بعد الاستخدام. الإرسال الفعلي للواتساب يتم عبر Edge Function `send-whatsapp-otp` التي تستخدم `service_role` (لا يُسرّب للعميل). راجع `docs/AUTH_SETUP.md`.

## 📋 قائمة تحقق أمنية قبل النشر

- [ ] ترقية سياسات RLS الإدارية لتعتمد على `role >= 2` (البند 1)
- [ ] إضافة سياسات UPDATE للمشرف على `payments`/`reports`/`deals` (البند 2)
- [ ] نقل إنشاء الإشعارات إلى RPC `SECURITY DEFINER` (البند 3)
- [ ] إنشاء keystore إنتاج + `key.properties` (غير مرفوع)
- [x] التأكد أن `verify_otp` يطبّق حد محاولات + انتهاء صلاحية → ✅ `generate_otp_v2` (5 طلبات/10د) + `verify_otp_v2` (5 دقائق صلاحية + حذف بعد الاستخدام)
- [ ] مراجعة bucket `offer_images`: عام للقراءة، والكتابة للمستخدمين المسجّلين فقط
- [ ] تدوير (rotate) أي توكن GitHub استُخدم أثناء التطوير
- [ ] تفعيل Email/Rate limiting في إعدادات مشروع Supabase

## 🔑 ملاحظة عن مفتاح anon
مفتاح `anon` **ليس سرّاً**؛ وجوده في تطبيق موبايل أمر طبيعي وآمن طالما أن RLS مضبوط. الحماية الحقيقية تأتي من **سياسات RLS**، وليس من إخفاء المفتاح. لذلك التركيز يجب أن يكون على البنود أعلاه.
