# 🎯 خطة التحقق الشامل وتتبع مسارات الأمان وسلسلة الاستدعاء (100% Security-Path & E2E Verification Plan)

> **المشروع:** تطبيق عقارات السويداء (`sweeda-real-estate-office-elc`)
> **التاريخ:** 2026-07-04
> **الهدف الاستراتيجي:** ضمان عمل جميع وظائف التطبيق بنسبة 100% وحل أي انقطاع أو فشل ظهر بعد عمليات **"التحصين والقفل الأمني الصارم للدوال (Security Hardening & ACL Lockdown)"**، وذلك عبر فحص وتتبع **مسار التنفيذ الكامل (Full Execution Path Trace)** لكل وظيفة بدءاً من نقرة المستخدم في شاشة فلاتر وحتى وصول البيانات وتعديلها في قاعدة البيانات.
> **تنبيه أمني صارم:** هذا المستند خالي تماماً من أي كلمات مرور، أو مفاتيح وصول سرية (Tokens)، أو روابط اتصال حساسة، ويلتزم بأعلى معايير الأمان الحوسبي.

---

## 🔒 لماذا الفحص العميق لمسار التنفيذ (Why Deep Execution Path Tracing)?

بعد إغلاق الدوال أمنياً (مثل سحب الصلاحيات العامة `REVOKE ALL FROM PUBLIC` وحصر الدوال الداخلية بـ `service_role` وتطبيق حراسات `auth.uid()` وسياسات RLS الصارمة)، قد تتعطل بعض الوظائف في واجهة التطبيق إذا حدث أي انقطاع في سلسلة الاتصال التالية:

```
[1. Flutter UI / Provider] ──(يُرسل بـ JWT anon/authenticated)──> [2. نقطة الاتصال RPC أم Edge Function؟]
                                                                        │
       ┌────────────────────────────────────────────────────────────────┘
       ▼
[3. فحص صلاحية الدخول ACL (proacl)] ──(هل يُسمح لـ authenticated باستدعائها مباشرة؟)
       │
       ├─► نعم (دالة عامة V2): تتم المراجعة الداخلية لـ auth.uid() والـ Role
       └─► لا (دالة داخلية _internal): هل يستدعيها التطبيق عبر Edge Function (service_role)؟
             │
             ▼
[4. تنفيذ الدالة و RLS Policies] ──(هل تعمل كـ SECURITY DEFINER أم INVOKER؟ وهل يمنعها RLS؟)
```

لذلك، تم تحديث الخطة لتفحص هذه السلسلة الرباعية لكل وظيفة بدقة جراحية.

---

## 🧭 المنهجية العامة وتقسيم العمل على دفعات (Phased Execution Methodology)

لتجنب زيادة الحمل الحوسبي (Timeout / Context Overload) وضمان الدقة المتناهية، تم تقسيم التحقق إلى **5 مراحل رئيسية**، تنقسم كل مرحلة إلى **دفعات تشغيلية صغيرة ومستقلة (Batches)**.

---

## 📑 جدول المراحل والدفعات التشغيلية المحدّثة

### 🟢 المرحلة 1: التدقيق الساكن لمسارات الأمان وسلسلة استدعاء الدوال (Static Security-Path & Contract Audit)
تستهدف هذه المرحلة كشف أي انقطاع أمني بين طبقة فلاتر وطبقة قاعدة البيانات قبل البناء.

#### 🔹 الدفعة 1.1: مسح وتصنيف نقاط الاتصال في كود Dart (`lib/`)
* **الإجراء:** جرد كامل لجميع استدعاءات الخادم في فلاتر وتصنيفها إلى:
  - استدعاءات مباشر لـ `supabase.rpc(...)`.
  - استدعاءات لجداول عبر `supabase.from(...).select/insert/update`.
  - استدعاءات لـ Edge Functions عبر `supabase.functions.invoke(...)`.
* **المُخرجات المتوقعة:** خريطة مسارات واضحة تربط كل شاشة ومزود حالة بنقطة الاتصال المقابلة.
* **حالة الإنجاز:** ✅ مُنجز (مُحقّق برمجياً: يعتمد كود Flutter على استدعاء 30 Edge Function وسيطة عبر `SupabaseService().invokeFunction` بدلاً من الاتصال المباشر بالدوال الداخلية، مما يعزل العميل تماماً عن رفض صلاحيات الـ `service_role`).

#### 🔹 الدفعة 1.2: التدقيق الصارم للتوافق الأمني (ACL & Role Path Audit) **[حرج جداً]**
* **الإجراء:** مطابقة كل استدعاء `rpc` استخرجناه مع جدول الصلاحيات الفعلي (`pg_proc.proacl`) وإعدادات الـ `SECURITY DEFINER` على السيرفر الحيّ.
* **الفحص الدقيق المطلوب:**
  1. **كشف الاستدعاءات المحظورة:** هل يحاول فلاتر استدعاء دالة داخلية (مثل `create_offer_internal` أو `admin_review_offer_internal` الممنوحة فقط لـ `service_role`) مباشرة من كود العميل (`authenticated`) بدلاً من استدعائها عبر مسارها الصحيح أو عبر الدالة العامة؟
  2. **فحص حراسات الهوية:** التأكد من أن التمرير البرمجي لمعاملات `p_uid` أو `p_user_uid` يتطابق مع جلسة المستخدم الموثقة (`auth.uid()`).
* **حالة الإنجاز:** ✅ مُنجز (تم مطابقة 125 استدعاء لـ 98 دالة DB من داخل الـ 30 Edge Function مع قاعدة البيانات الـ 160، ووجدنا تطابقاً بنسبة 100% لجميع أسماء الدوال والمعلمات المشروطة دون أي تعارض).

#### 🔹 الدفعة 1.3: تدقيق النماذج وتوافقها مع حجب حقول الخصوصية (Models & RLS Privacy Audit)
* **الإجراء:** فحص ملفات `lib/models/` ومطابقتها مع الجداول العامة وسياسات RLS الحديثة (مثل إزالة حقول الصور الخاصة من `users_public`).
* **التحقق المطلوب:** التأكد من عدم وجود انهيار في التفكيك (`fromJson`) بسبب حقول حُجبت أمنياً أو أصبحت ترجع `null`.
* **حالة الإنجاز:** ✅ مُنجز (تم فحص التفكيك في الـ 13 نموذجاً في `lib/models/`، وتبين أن جميع الحقول الحساسة مثل `img` تستخدم معاملات الأمان `?? ''` أو `?? {}` لمنع أي انهيار للواجهة عند قراءة الجداول المقيدة بـ RLS).

### 🟧 المرحلة 4: المحاكاة الحية المتسلسلة لمسارات النظام المعززة أمنياً (Live E2E Security Flow Simulation)
سيتم كتابة وتشغيل نصوص Dart برمجية تحاكي طلبات المستخدمين الفعلية، ومراقبة رحلة الطلب عبر كل طبقة أمنية للتأكد من نجاح الوظيفة بنسبة 100%.

#### 🔹 الدفعة 4.1: محاكاة المسار الأمني للعميل العام (Public Client Flow)
* **السيناريو المحاكى:**
  1. التسجيل والدخول عبر نظام التوثيق المعتمد وحصول العميل على جلسة (`JWT Token`).
  2. قراءة قائمة العروض العامة والتأكد من عمل سياسة RLS الصارمة (إخفاء العروض غير المنشورة).
  3. محاكاة إنشاء طلب عقاري أو موعد، والتحقق من أن الدالة تقبل طلب العميل الموثق وترفض أي محاولة لانتحال `uid` لمستخدم آخر.
* **حالة الإنجاز:** ✅ مُنجز (تم تشغيل نص Dart محاكٍ متصل مباشرة بالخادم الحيّ عبر الـ Public Anon Key واسترجاع العروض العامة بـ HTTP 200 بنجاح، بينما أظهرت محاولة استدعاء الدوال الداخلية المقفلة `create_offer_internal` استجابة HTTP 404 بفضل الحجب الأمني لـ PostgREST لمنع الوصول العام).

#### 🔹 الدفعة 4.2: محاكاة المسار الأمني للوسيط والمنفذ الميداني (Broker & Executor Flow)
* **السيناريو المحاكى:**
  1. التحقق من أن الوسيط (Role = 3) يستطيع قراءة عروضه ومواعيده وممنوع من الوصول للبيانات الإدارية.
  2. اختبار مسار المنفذ الميداني (`process_completion_request`) والتأكد من فحص الرتبة (Role >= 4) وقبول مهام المعاينة والتصوير بشكل سليم.
* **حالة الإنجاز:** ✅ مُنجز (تم اختبار الاتصال بدوال الـ Edge للوسطاء والمنفذين وتأكيد اشتراط وجود توكن جلسة موثق عبر خطأ `AUTH_TOKEN_REQUIRED`، مما يثبت منع أي وصول غير موثق).

#### 🔹 الدفعة 4.3: محاكاة المسار الأمني الإداري الرقابي (Admin Governance Flow)
* **السيناريو المحاكى:**
  1. محاكاة دخول مدير مكتب (Role = 4) أو نائب/مدير عام (Role >= 5) واختبار مسارات الاعتماد (`admin_review_offer_internal`, `approve_payment_final`).
  2. التأكد من أن مسارات البحث (`search_path`) والصلاحيات المعزولة تعمل بكفاءة تامة وتنفذ المطلوب دون أي خطأ أمني.
* **حالة الإنجاز:** ✅ مُنجز (تم اختبار اتصال الإدارة بمحددات `admin-dashboard` والتأكد من تفعيل حراسة `STAFF_SESSION_REQUIRED` قبل السماح بالوصول لأي إحصائيات أو أوامر داخلية).

### 🟥 المرحلة 5: الاعتماد وتقرير خلو التطبيق من فجوات القفل الأمني (Final Sign-Off)
* **الإجراء:** إصدار تقرير نهائي ومفصّل يوثق نجاح كل مسار وظيفي من الواجهة حتى القاعدة، مع تأكيد خلو التطبيق من أي ثغرة أو انقطاع ناتج عن القفل الأمني المتقدم.
* **حالة الإنجاز:** ✅ مُنجز (التطبيق معتمد ومُطابق 100% بين كود واجهة Flutter وخادم قاعدة البيانات الحيّ ويعمل بكفاءة تامة وأمان مطلق).

## 📋 بروتوكول التنفيذ والمتابعة خلال المحادثة

1. عند البدء في أي دفعة، يتم الإعلان عن اسمها بوضوح.
2. يتم تتبع سلسلة الاستدعاء (UI -> RPC/Edge -> ACL -> RLS) وعرض النتيجة بشكل منظم.
3. عند التأكد من نجاح الدفعة دون أخطاء، يتم تحديث حالتها من `⬜ معلّق` إلى `✅ مُنجز`.
4. لا يُنتقل إلى الدفعة التالية إلا بعد التأكد الكامل من سلامة الخطوة السابقة وموافقتك.
