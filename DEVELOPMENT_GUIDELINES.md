# 🛠️ دليل تطوير مشروع "عقارات السويداء"
> هذا المستند هو مرجع إلزامي للمطور (الوكيل الذكي) لضمان استمرارية الجودة وتطابق الكود مع التوثيق.

## 🔄 دورة حياة التعديل (Workflow)
يُمنع إغلاق أي مهمة أو إبلاغ المستخدم بالانتهاء إلا بعد استكمال الخطوات الثلاث التالية بالترتيب:

### 1️⃣ المرحلة البرمجية (Coding Phase)
- تنفيذ التعديلات المطلوبة في ملفات `.dart` أو `sql`.
- التأكد من عدم وجود أخطاء برمجية (Syntax Errors).
- اختبار المنطق (Business Logic) لضمان عدم وجود ثغرات (مثل ثغرات العملات أو النقاط).

### 2️⃣ المرحلة التوثيقية (Documentation Phase) ⚠️ **(إلزامية)**
- مراجعة ملف `docs/SPEC.md`: إذا تم تغيير أي حقل في قاعدة البيانات أو إضافة وظيفة جديدة، يجب تحديث المواصفات فوراً.
- مراجعة ملف `docs/FEATURES_AUDIT.md`: نقل الميزة من "ناقصة/جزئية" إلى "كاملة" بعد التأكد من تنفيذها.
- تحديث أي ملفات مرجعية أخرى (مثل خطط الدفع أو الإشعارات).

### 3️⃣ مرحلة الرفع (Deployment Phase) ⚠️ **(إلزامية)**
- تنفيذ سلسلة أوامر الرفع لضمان وصول التعديلات للمستودع السحابي:
  ```bash
  git add .
  git commit -m "وصف دقيق للتعديلات"
  git push origin main
  ```

---

## 🚫 محظورات تقنية
- **لا** تترك أي `print` أو `debugPrint` في الكود النهائي.
- **لا** تضف نصوصاً ثابتة (Hardcoded)؛ كل شيء يجب أن يمر عبر `ConfigModel`.
- **لا** تغير في بنية الجداول دون تحديث ملف `setup.sql` في مجلد `supabase`.
