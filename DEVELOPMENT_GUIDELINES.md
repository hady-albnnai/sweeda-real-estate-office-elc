# 🛠️ دليل تطوير مشروع "عقارات السويداء"
> هذا المستند هو مرجع إلزامي للمطور (الوكيل الذكي) لضمان استمرارية الجودة وتطابق الكود مع التوثيق.

## 🔄 دورة حياة التعديل (Workflow)
يُمنع إغلاق أي مهمة أو إبلاغ المستخدم بالانتهاء إلا بعد استكمال الخطوات الثلاث التالية بالترتيب:

### 1️⃣ المرحلة البرمجية (Coding Phase)
- تنفيذ التعديلات المطلوبة في ملفات `.dart` أو `sql`.
- التأكد من عدم وجود أخطاء برمجية (Syntax Errors).
- اختبار المنطق (Business Logic) لضمان عدم وجود ثغرات (مثل ثغرات العملات أو النقاط).

### 2️⃣ المرحلة التوثيقية (Documentation Phase) ⚠️ **(إلزامية)**
- مراجعة ملف `docs/SPEC.md`: إذا تم تغيير أي حقل في قاعدة البيانات أو إضافة وظيفة جديدة، يجب تحديث المواصفات فوراً.
- مراجعة ملف `docs/FEATURES_AUDIT.md`: نقل الميزة من "ناقصة/جزئية" إلى "كاملة" بعد التأكد من تنفيذها.
- تحديث أي ملفات مرجعية أخرى (مثل خطط الدفع أو الإشعارات).

### 3️⃣ مرحلة الرفع (Deployment Phase) ⚠️ **(إلزامية)**
- تنفيذ سلسلة أوامر الرفع لضمان وصول التعديلات للمستودع السحابي:
  ```bash
  git add .
  git commit -m "وصف دقيق للتعديلات"
  git push origin main
  ```

## 🚫 محظورات تقنية
- **لا** تترك أي `print` أو `debugPrint` في الكود النهائي.
- **لا** تضف نصوصاً ثابتة (Hardcoded)؛ كل شيء يجب أن يمر عبر `ConfigModel`.
- **لا** تغير في بنية الجداول دون تحديث ملف `setup.sql` في مجلد `supabase`.

## 🎯 مبدأ الدقة المتناهية (إلزامي — أُضيف 2026-07-02 بتوجيه من صاحب المشروع)

كل مهمة تُنفَّذ بدقة متناهية وتحليل كامل مهما أخذت من وقت وجهد، **دون اختصار ودون حلول جزئية**. السببان الموجبان:

1. **سبب يخص المشروع:** التطبيق في مراحله النهائية قبل النشر — أي خطأ يصل للإنتاج
   يصيب مستخدمين حقيقيين، فلا مجال لتعديلات متسرعة أو غير مكتملة.
2. **سبب يخص كفاءة العمل:** الحلول الجزئية والاختصارات تجبر صاحب المشروع على فتح
   محادثات جديدة لنفس الموضوع وإعادة شرح السياق من الصفر — وهذا استنزاف للموارد
   والوقت دون داعٍ. الحل الكامل من أول مرة أوفر دائماً.

### قواعد التطبيق العملي لهذا المبدأ:
- **لا افتراضات عن السيرفر:** ملفات المستودع (migrations / FUNCTIONS_REFERENCE) ليست
  بالضرورة مطابقة لحالة السيرفر الفعلية. قبل أي تعديل يمس قاعدة البيانات، يُتحقق من
  الواقع عبر استعلامات حية (تواقيع الدوال، الأعمدة، الصلاحيات، البيانات الفعلية).
- **تتبع كامل للتبعيات:** عند تعديل أي دالة أو نوع إرجاعها، تُحصر كل الجهات المستدعية
  (دوال SQL أخرى، Edge Functions، Providers، الشاشات) وتُحدَّث جميعها في نفس المهمة.
- **معالجة جذر المشكلة لا عرضها:** إن كشف التحليل ثغرات مجاورة في نفس النطاق،
  تُذكر صراحة في التقرير حتى لو لم تُصلَّح فوراً — لا تُترك مكتومة.
- **لا تنفيذ دون تصريح:** أي تعديل (كود، رفع، تطبيق على السيرفر) يتطلب موافقة صريحة
  من صاحب المشروع، مع تقرير واضح بما سيُنفَّذ قبل التنفيذ.
- **الإنجاز الحقيقي فقط:** لا يُعلن اكتمال ميزة إلا بعد اكتمال سلسلتها كاملة
  (سيرفر + Edge + Provider + واجهة + توثيق + رفع) وتحديد ما تبقى على عاتق
  صاحب المشروع (تنفيذ SQL، نشر Functions، إعدادات تشغيلية) بشكل صريح.
